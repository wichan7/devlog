최근 개인 프로젝트로 Next.js를 꽤 사용해 보았다. 왜 Next였냐 하면, 우선 "Next는 React의 미래다, 모르면 도태된다"며 겁을 주는 말들이 신경 쓰였고, 실제로 React 팀에서 Next.js를 표준으로 밀고 있으니 알고는 있어야겠다는 생각이었다.  

그리고 직접 경험해 본 Next.js는 혼란스러운 부분이 많은 도구었다.  

## SSR과 서버의 실체
Next.js 관련 글을 보면 대부분 SSR과 Hydration을 굉장히 큰 장점으로 꼽는다. 여기서 첫 번째 혼란이 왔다. React는 단순히 정적 자원을 생성하는 도구인데 대체 어떻게 서버 사이드 렌더링을 한다는 건가?

알고 보니 Next는 빌드 시 Node.js 기반의 서버가 함께 빌드된다. 이 Node 서버가 Hydration 등의 작업을 수행해 주는 구조다. 인프라 관점에서 보면 Nginx 같은 웹 서버만 필요했던 React에 비해, 중간에 WAS를 관리해야 하는 부담이 추가된 셈이다. 후술할 Vercel 같은 플랫폼을 쓰지 않는다면 이것의 관리가 상당히 큰 비용이 될 것이다.

## 서버와 클라이언트의 모호한 경계
Server Component와 Client Component의 구분은 혼돈의 도가니탕이었다.

SSR이 장점이라니까 최대한 Server Component 위주로 소스를 짜봤다. 테마 적용을 위해 zustand 전역 store를 사용했는데 아뿔싸 Server-Side JSX와 Client-Side JSX는 상태를 공유할 수 없었다.  

일단 Java + Thymeleaf 같은 조합은 사용하는 언어가 다르니 서버/클라이언트 구분이 자연스러워 이런 이해가 필요하지 않았다. 하지만 Next는 모든 소스가 JS다. 코드를 짜면서도 이 로직이 서버에서 실행되는지 클라이언트에서 실행되는지 항상 의식해야 했다.  

이 방식은 BE와 FE의 책임을 나누어 발전해 온 시대상을 역행한다는 느낌을 받았다.

결국 장점이라는 서버 컴포넌트는 다 걷어내고 'use client'를 도배하게 되었으며, SSR 대신 /api 하위에 API를 만들고 호출하는 익숙한 구조로 돌아가 버렸다.


## Server Action 이라는 이름의 RFC
웹 버전 RFC인 Server Action이다. 서버 함수를 클라이언트에서 import해 쓰는 방식인데 직렬화 가능한 파라미터와 리턴이면 'use client' 내부에서도 함수를 가져다 쓸 수 있다.  

아니 server-side 소스코드는 서버에서만 실행된다며? 내부적으로는 액션 함수를 만들면 next가 해당 함수에 고유값을 만들고, 클라이언트 소스코드에도 해당 함수를 http call의 형태로 실행하고 결과를 return받도록 빌드된다.  

이게 최근 보안 취약점(CVE) 10점 만점을 기록했다. 서버 액션을 Promise를 사용해 직렬화/역직렬화하는 부분에서 구멍이 발생했고, 결과적으로 페이로드에 `.then()` 함수를 잘 인젝션하면 Node 서버에서 쉘까지 실행할 수 있는 수준이었다. 시대를 역행한 대가를 톡톡히 받은 셈이다.


## 나는 Next가 싫다
나는 Next가 싫다. 하지만 내세울 대안은 없다.

신명나게 깠지만 React 베이스의 개발자가 SEO를 위한 SSR을 챙기면서 이 정도의 개발 경험을 얻는 대체재가 뭐가 있을까?

Astro? Remix? 안해봐서 솔직히 잘 모르겠다. 맛봤던 Thymeleaf, JSP랑 비교하자면 나는 Next 쓸 것 같다.

그리고 Next는 흠..좀.. 이지만 Vercel은 짱이다. Next 프로젝트를 만들면 Vercel이 제공하는 Rich한 기능이 어마어마하다. GitHub 레포에 연동, CI/CD, 인증서 관리, Analytics, Serverless-RDB, No-SQL DB까지 대부분 원클릭, 그리고 무료로 구성된다. 

인프라 구축 공수를 스킵하고 기능 구현에만 집중할 수 있다는 건 재미 원툴의 개인 프로젝트에서 굉~장히 큰 장점이다.
 

## 결론
흔한 개발 격언처럼 세상에 은탄환은 없다. 적어도 내가 보기에 Next.js는 오해하기 쉬운 개념과 보안 취약점 덩어리였다.  

기술이 비대하게 느껴졌고, 명성에 비해 실망스러운 부분이 많았다. Vercel이라는 자본력 있는 강결합된 플랫폼에 힘입어 은탄환처럼 마케팅 되고 있는 것처럼 보인다.

근데 사실 이 블로그도 Next로 개발했다. Vercel의 배포 편의성은 진짜 짱이다.  

서버 측 마법이 너무 많아서 엔터프라이즈급으로 추천하기는 힘들겠지만, DX를 잘 잡은 Vercel + Next가, MVP 같은 소규모 개발 시장에서는 앞으로도 승승장구하지 않을까?
